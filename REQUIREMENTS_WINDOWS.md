# プロジェクト名: UltraLight TC Player for Windows（仮）

## 1. ビジョン（主語と目的）

- **主語**: 開発者本人（自分専用のプロツール）
- **目的**: Windows上で軽快に動作しつつ、H.264のmp4動画を主対象に、IN / OUT位置の `hh:mm:ss:ff` を確認してffmpeg切り出しコマンドを作れる環境を構築する。
- **方向性**: macOS版で想定していた「QuickTime Player相当の軽快さ・シンプルさ・視認性」を、Windowsネイティブ環境へ置き換える。

## 2. 対応OS・実行環境

- **対応OS**: Windows 11を主対象とする。
- **将来的な対応候補**: Windows 10は必須対象外。ただし、実装方式に大きな追加コストが発生しない場合のみ検討する。
- **配布形態**:
  - まずは自分専用ツールとしてローカル実行できることを優先する。
  - Microsoft Store配布は初期要件に含めない。
  - 一般配布は想定せず、個人利用としてローカルで起動できることを優先する。

## 3. 核心的な要件（Must have）

### 3.1 パフォーマンス

- Windowsネイティブの動画再生基盤を利用し、CPU / メモリ負荷を最小限に抑える。
- H.264のmp4再生を主対象とする。
- H.265 / HEVC再生は必須要件ではなく、有料ライブラリや有料コーデックを前提にしない。
- 再生処理は、WindowsのGPU支援・ハードウェアアクセラレーションに乗せることを優先する。
- Electronなどの重い実行基盤は採用しない。

### 3.2 UI / 表示機能

- **タイムコード表示**:
  - `hh:mm:ss:ff`（時:分:秒:フレーム）を常時表示する。
  - 現在位置と動画のフル尺を `現在位置 / フル尺` の形式で表示する。
  - 放送用途の厳密なタイムコードではなく、プレイヤーの現在位置とFPSから算出する確認用表示とする。
  - 再生中も数字の桁位置が揺れないことを重視する。
- **フレーム単位の操作**:
  - 左右キーで目安としての1フレーム送り / 戻しを行う。
  - 厳密なフレーム精度よりも、軽く確認できる操作感を優先する。
- **ドラッグ＆ドロップ**:
  - 動画ファイルをウィンドウにドロップしたら即読み込み・再生できること。
- **IN / OUT取得**:
  - `I` キーで現在位置をINとして記録する。
  - `O` キーで現在位置をOUTとして記録する。
  - IN / OUTを `hh:mm:ss:ff` で表示する。
- **ffmpegコマンド生成**:
  - IN / OUTが揃ったら、ffmpeg切り出しコマンドを生成する。
  - コマンド欄は常時表示する。
  - コマンドはコピーできればよく、MVPではアプリ内でffmpegを実行しない。
- **最小限のコントロール**:
  - 再生 / 一時停止
  - シークバー
  - 現在位置 / フル尺 / IN / OUTのタイムコード表示
  - 音量スライダー
  - ミュートボタン
  - 再生 / 一時停止ボタンとミュートボタンは、表示文字が変化してもサイズが変わらない固定幅とする。

### 3.3 FPS自動判別と同期

- 動画ファイル読み込み時に、メタデータから可能な限り正確なフレームレートを取得する。
- 以下の一般的なFPSを自動識別する。
  - 23.976 fps
  - 24 fps
  - 25 fps
  - 29.97 fps
  - 30 fps
  - 59.94 fps
  - 60 fps
- 識別したFPSに基づき、タイムコード上の「1フレーム」の長さを動的に計算する。
- VFR（可変フレームレート）動画の場合は、初期版では厳密対応を必須としない。ただし、誤差が出る可能性をUIまたはREADMEに明記する。

### 3.4 キーボードショートカット

- `Space`: 再生 / 一時停止のトグル
- `Right Arrow`: 1フレーム送り
- `Left Arrow`: 1フレーム戻し
- `Ctrl + O`: ファイルを開く
- `I`: 現在位置をINに設定
- `O`: 現在位置をOUTに設定
- `Esc`: 選択中のINまたはOUTをクリア
- `Delete`: IN / OUTを両方クリア
- 1フレームの定義は、自動判別された動画のFPSに準拠する。

## 4. Windows版の推奨技術方針

### 4.1 第一候補: Windowsネイティブ構成

- **UI候補**:
  - WinUI 3
  - WPF
  - Windows App SDK
- **動画再生基盤候補**:
  - Media Foundation
  - Direct3D連携
  - Windows標準のメディア再生コンポーネント
- **開発言語候補**:
  - C#
  - C++ / WinRT

初期開発では、実装難度とメンテナンス性を考慮し、C# + WPFを第一候補とする。

### 4.2 避けたい構成

- Electronベースの実装
  - 起動・メモリ・配布サイズが重くなりやすいため、今回の「軽量・自分専用プロツール」という目的と合いにくい。
- WebView中心の動画プレーヤー
  - フレーム単位の操作や軽量なローカルツールという目的に合いにくいため、初期候補から外す。
- 汎用プレーヤーの単なるラッパー
  - VLCやmpvのラッパーは実装速度では有利だが、「Windowsネイティブで軽い専用ツール」という目的からは優先度を下げる。

## 5. 非機能要件（こだわりの「型」）

### 5.1 シンプルさ

- プレイリストは不要。
- 複雑な設定画面は不要。
- 起動後、動画を開くかドロップしたらすぐ使えること。
- 操作体系は最小限にする。

### 5.2 軽量性

- 起動が速いこと。
- 余計な常駐処理を持たないこと。
- 動画再生中のCPU使用率をできるだけ低く保つこと。
- H.264のmp4再生で軽快に動作することを優先する。

### 5.3 レスポンス性

- フレーム送り時のプレビュー描画遅延を極力抑える。
- 左右キーを押した瞬間に、目安として1フレーム分移動したことが分かる操作感を目指す。
- シークバー操作時も、可能な限り即座にプレビューが更新されること。
- シークバーの任意の位置をクリックした時、その位置へ直接シークできること。
- シークバーのつまみによるドラッグ操作も維持すること。

### 5.4 UIスタイル

- Windows標準アプリらしいシンプルな外観を基本とする。
- 映像表示領域を主役にし、コントロール類は最小限に抑える。
- タイムコード、FPS、IN / OUT、クリア操作が狭いウィンドウ幅でも重ならないレイアウトとする。
- 必要に応じてタイムコード情報とIN / OUT操作を段分けし、各情報の視認性を維持する。
- 再生コントロールは、マウスホバー時のみ表示されるフローティングスタイルを目指す。
- タイトルバーは通常表示でもよいが、将来的にはカスタムタイトルバーやボーダーレス表示も検討する。
- QuickTime Playerをそのままコピーするのではなく、「軽く、映像が主役で、操作が少ない」という思想をWindows風に置き換える。

### 5.5 フォント構成

- **メインUI表示**:
  - Windows標準の `Segoe UI` を基本とする。
  - 日本語環境では `Yu Gothic UI` も候補とする。
- **タイムコード表示**:
  - 数字の視認性を最優先し、等幅数字または等幅フォントを使用する。
  - 候補:
    - `Cascadia Mono`
    - `Consolas`
    - `Segoe UI` の等幅数字指定が可能な場合はそれを利用する。
  - 再生中に数字が横に揺れないことを必須とする。

## 6. タイムコード計算方針

### 6.1 基本方針

- 現在の再生位置を秒単位で取得する。
- 読み取ったFPSを用いて、現在の総フレーム数を算出する。
- 総フレーム数から `hh:mm:ss:ff` を算出する。

### 6.2 計算イメージ

```text
totalFrames = round(currentSeconds * fps)
framesPerHour = fps * 60 * 60
framesPerMinute = fps * 60

hour = totalFrames / framesPerHour
minute = (totalFrames % framesPerHour) / framesPerMinute
second = (totalFrames % framesPerMinute) / fps
frame = totalFrames % fps
```

### 6.3 29.97 / 59.94 fpsについて

- 初期版ではノンドロップフレーム形式の `hh:mm:ss:ff` を基本とする。
- 放送用途などでドロップフレーム表記が必要になった場合は、後続バージョンで検討する。
- READMEには、初期版のタイムコードが「確認用のフレーム表示」であり、ドロップフレーム表記を厳密に保証するものではないことを明記する。

### 6.4 ffmpeg向け時間形式

- 画面表示は `hh:mm:ss:ff` を使用する。
- ffmpegコマンドでは、同じ位置を `hh:mm:ss.fff` に変換して使用する。
- 例:

```text
画面表示用: 00:01:12:08
ffmpeg用:   00:01:12.267
```

## 7. フレーム送り / 戻しの実装方針

- 1フレームの秒数を `1 / fps` として計算する。
- `Right Arrow` 押下時:
  - 現在位置 + 1フレーム分へシークする。
- `Left Arrow` 押下時:
  - 現在位置 - 1フレーム分へシークする。
- シーク後は即座にプレビューを更新する。
- フレーム送り / 戻しは厳密なデコードフレーム単位ではなく、`1 / fps` 秒ぶんシークする目安操作として扱う。
- 実装の安定性を優先し、初期版ではフレーム操作時に一時停止してから移動してよい。

## 8. IN / OUTとffmpegコマンド生成

- `I` キーで現在位置をINに設定する。
- `O` キーで現在位置をOUTに設定する。
- `I` または `O` で設定した側を選択中として扱う。
- `Esc` キーで選択中のINまたはOUTだけをクリアする。
- `Delete` キーでIN / OUTを両方クリアする。
- UIには `Clear In` / `Clear Out` / `Clear All` ボタンを置く。
- コマンド欄は常時表示する。
- IN / OUTが揃っていない場合、ffmpegコマンドは生成しない。
- IN / OUTが揃った時だけ、コマンドをコピー可能にする。
- OUTがIN以前の場合、コマンドは生成せずエラー状態を表示する。
- 入力ファイルパスは、現在プレイヤーで再生しているソースファイルのフルパスを使用する。
- 出力ファイル名は、同じフォルダに `source-cut.ext` の形式で動的に生成する。
  - 例: `sample.mp4` -> `sample-cut.mp4`
  - 例: `sample.mov` -> `sample-cut.mov`
- MVPでは高速切り出し用に `-c copy` のコマンドを生成する。
- `-c copy` は高速・無劣化だが、キーフレーム位置の都合で切り出し位置が表示位置と少しずれる可能性があることをREADMEに明記する。
- コマンド例:

```powershell
ffmpeg -ss 00:00:10.000 -to 00:00:25.000 -i "C:\Videos\sample.mp4" -c copy "C:\Videos\sample-cut.mp4"
```

## 9. 対応ファイル形式

### 9.1 初期対応

- `.mp4`
- `.mov`
- `.m4v`

### 9.2 コーデック

- H.264のmp4を最優先対象とする。
- H.265 / HEVCは必須要件ではない。
- HEVC再生には、Windows環境側のHEVCデコーダーやGPU / ドライバー状態の影響を受ける可能性がある。
- 有料ライブラリや有料コーデックを前提にした実装は初期版では採用しない。

## 10. 初期版でやらないこと

- プレイリスト機能
- 動画変換機能
- 字幕編集機能
- 波形表示
- 複数動画の比較表示
- カラーグレーディング機能
- 書き出し機能
- ネットワーク再生
- クラウド同期
- 詳細なメディア管理
- アプリ内でのffmpeg実行
- 有料ライブラリ前提のHEVC対応

## 11. 開発時の検証項目

- H.264のmp4動画が再生できるか。
- CPU使用率が高すぎないか。
- GPUデコードが有効になっているか。
- 23.976 / 29.97 / 59.94 fpsの動画でタイムコード表示が破綻しないか。
- 左右キーによる1フレーム送り / 戻しが期待通りに動作するか。
- `I` / `O` でIN / OUTを記録できるか。
- `Esc` で選択中のINまたはOUTだけをクリアできるか。
- `Delete` でIN / OUTを両方クリアできるか。
- `Clear In` / `Clear Out` / `Clear All` が期待通りに動作するか。
- IN / OUTが揃った時だけffmpegコマンドが生成され、コピーできるか。
- 出力ファイル名が `source-cut.ext` として動的に生成されるか。
- 音量スライダーとミュートボタンが動作するか。
- Play / PauseおよびMute / Mutedの表示切り替え時にボタンサイズが変化しないか。
- 長尺動画でシークが重くならないか。
- シークバーをクリックすると、その位置へ直接移動できるか。
- シークバーのつまみをドラッグしてシークできるか。
- ドラッグ＆ドロップで即読み込みできるか。
- 数字表示が再生中に横揺れしないか。
- 現在位置とフル尺が `現在位置 / フル尺` 形式で表示されるか。
- タイムコードの区切り文字や隣接するUIがタイムコード表示へ重ならないか。
- ウィンドウサイズ変更時に映像とコントロールが崩れないか。

## 12. Codex / AI実装依頼時の注意

- 元のMac版要件に含まれていた `AVFoundation`、`Swift`、`AppKit`、`SwiftUI`、`QuickTime Player完全模倣`、`Hiragino Sans` はWindows版では前提にしない。
- Windows版では、`Media Foundation`、`Direct3D`、`WPF`、`Segoe UI`、`Cascadia Mono` などを候補として検討する。
- 初期実装はC# + WPFを優先する。
- ただし、実装難度が高くなる場合は、まず「軽量に起動し、H.264 mp4を再生し、タイムコードを表示し、IN / OUTを取得し、ffmpegコマンドをコピーできる」ことを優先する。
- UIの完成度よりも、初期版では動画確認とIN / OUT取得の実用性を優先する。
- 実装方式の選定では、以下の優先順位を守る。
  1. 軽量性
  2. H.264 mp4再生の安定性
  3. IN / OUT取得とffmpegコマンド生成の実用性
  4. タイムコード表示の視認性
  5. UIの美しさ

- `I` / `O` でIN / OUTを記録できる。
- `Esc` で選択中のINまたはOUTをクリアできる。
